# 全 AI 开发环境构建哲学

> **AI 时代的软件工程 = 传统工程实践 + AI 可读的规范体系 + AI 行为的一致性保障 + AI 产出的质量管控**

本文是 AIES 脚手架的**指导思想**，回答一个核心问题：
**当 AI 成为团队"主力开发者"时，工程结构应该长什么样？**

---

## 一、核心判断：AI 是"有超强记忆力但极端健忘"的实习生

把 AI 当成一个这样的同事：

| 特征 | 意味着什么 |
|------|-----------|
| 📚 **超强知识面** | 不需要教框架、语言、API 细节 |
| 🧠 **超强上下文窗口** | 一次能读几千行代码 |
| 💀 **会话级失忆** | 关掉窗口就忘记昨天做了什么 |
| 🤥 **自信地编造** | 不知道的事会编出看起来合理的方案 |
| 🎯 **极度依赖指令** | 给什么上下文就做什么事 |
| ⚡ **10 倍产能** | 代码量大，但盲区多 |

**结论**：工程不再需要「让 AI 学会这个项目」，而是要「让项目对 AI 友好」。

---

## 二、三层能力模型

全 AI 开发环境由三层能力堆叠而成，缺一不可：

```
┌─────────────────────────────────────────────┐
│  第 3 层：编排层（Orchestration）              │
│  - 多 Agent 流水线                            │
│  - Hooks / Slash Commands                     │
│  - CI / Git Hook 质量门                       │
│  价值：让 AI 自动化地完成复杂流程               │
├─────────────────────────────────────────────┤
│  第 2 层：记忆层（Memory）                     │
│  - 任务管理（task.json）                       │
│  - 会话日志（journal-N.md）                    │
│  - 变更日志（changelog.md）                    │
│  价值：跨会话不失忆、不重复犯错                  │
├─────────────────────────────────────────────┤
│  第 1 层：规范层（Spec / Rule）                │
│  - 架构约束、命名规范、代码风格                  │
│  - DTO 注释规范、错误码、日志规范               │
│  - 业务术语表、数据模型                        │
│  价值：让 AI 产出一致、可复用、可审查             │
└─────────────────────────────────────────────┘
```

**建议采用顺序**：第 1 层 → 第 2 层 → 第 3 层。**不要跳级**。

- 没有第 1 层就上多 Agent = 批量生产垃圾代码
- 没有第 2 层就上 Hooks = 每次会话都是新的人
- 没有第 3 层也能工作 = 多数团队的真实状态

## 三、八维方法论（个人守则，跨项目生效）

这是每次 AI 对话都应该自动生效的行为准则：

### 维度 1：规范先行 — 动手前先定规矩

新项目/新模块开始前，**先确认或创建项目 Spec**。AI 必须主动问「有 spec 吗？没有要不要先写」。

### 维度 2：代码给 AI 读 — 写 AI 能精确理解的代码

- 所有 DTO 字段必须有结构化注释（含义 + 枚举值 + 默认值 + 约束）
- API 接口必须有 OpenAPI/Swagger 描述
- 不依赖「隐含知识」，一切用代码和注释表达

### 维度 3：Rule 保一致 — 保证 AI 产出风格统一

- 每次生成代码前必须加载项目 Spec
- 发现产出风格不一致 → 立即纠正 + 补充 Spec
- 新约定必须沉淀到 Spec，不能只在对话中说一次

### 维度 4：索引自维护 — 保持项目地图可用

- 每次完成功能后更新 `.ai/index.md`
- 索引应包含：模块职责、路由表、数据模型、调用链、关键文件
- 新增文件必须在索引中体现

### 维度 5：可追溯性 — 记录 AI 参与了什么

- 关键决策处必须有「为什么这样做」注释
- commit message 带 `[ai-assisted]` 标记
- 复杂逻辑人必须理解每一行

### 维度 6：上下文管理 — 精准喂上下文

遵循「最小充分上下文」原则：**给够参考但不过载**。不要把整个项目丢给 AI。

### 维度 7：隐性技术债 — 审查 AI 代码的盲区

重点审查 AI 常见盲区：
- 并发安全、事务完整性、边界条件
- SQL 注入、越权、信息泄漏
- 硬编码、遗漏日志、未设置默认值

### 维度 8：Prompt 资产化 — 好的 Prompt 要沉淀

反复用到的 Prompt 保存为模板，放在 `.ai/prompts/` 下。

## 四、为什么「规范 + 日志 + 清单」这个组合最优

我们尝试过很多方案，最终留下这三件事，有它的道理：

### Spec（规范）→ 解决"一致性"

```
问题：AI 每次生成的代码风格都不一样
方案：把约定写成 Spec，AI 加载后产出自动对齐
关键点：Spec 不是给人读的文档，是给 AI 读的指令
```

### Journal（日志）→ 解决"失忆"

```
问题：新会话不知道上次做到哪了
方案：会话结束追加日志，下次读取接续上下文
关键点：日志要结构化（标题/commit/摘要/关键决策）
```

### Checklist（清单）→ 解决"盲区"

```
问题：AI 生成的代码看起来完美，实则漏洞百出
方案：列出 8 大盲区（架构/安全/事务/边界/错误/性能/日志/AI 盲区）逐项检查
关键点：清单必须由人过（AI 自检不可靠）
```

## 五、反模式（不要做这些事）

### ❌ 反模式 1：规范写给人读

```markdown
# ❌ 写给人的 Rule
我们的项目使用 Gin 框架，遵循 MVC 架构，请大家注意代码质量。

# ✅ 写给 AI 的 Spec
架构约束（强制）：Router → API → Service → Model 四层，严禁逆向依赖
Model 层函数必须首参数 ctx context.Context，必须 DB.WithContext(ctx)
```

### ❌ 反模式 2：一次性把所有上下文给 AI

```
❌ 把整个项目 40 个文件打包给 AI，指望它自己选要看哪个
✅ 按场景列出「新增 API 需要这 7 个文件」，AI 按需读取
```

### ❌ 反模式 3：没有 Spec 就上多 Agent

```
❌ 没有规范 → 用多 Agent → 每个 Agent 产出风格都不同 → 全是垃圾
✅ 先写 Spec 让单 Agent 产出一致 → 再考虑是否需要多 Agent
```

### ❌ 反模式 4：依赖 AI 自检通过就提交

```
❌ AI 说「代码无问题可以提交」就 commit
✅ AI 自检 + 人过 review-checklist.md + 跑 CI = 三道门
```

### ❌ 反模式 5：每个平台维护一份 Rule

```
❌ .cursorrules / CLAUDE.md / AGENTS.md 各写一份，越改越不一致
✅ 规范写在 .aies/spec/ 一份，各平台入口文件只做"引用"
```

### ❌ 反模式 6：Rule 写得太抽象

```markdown
# ❌ 抽象规则
请遵守 SOLID 原则

# ✅ 可执行规则
函数参数不超过 4 个，第一个必须是 ctx context.Context
超过则封装为 struct 放 internal/entity/
```

## 六、团队协作模式

### 模式 A：单人 + AI（最常见）

- 规范在仓库，会话日志可提交
- 自己就是审查者，清单过一遍即可
- 适合 80% 的团队

### 模式 B：小团队（2-5 人）+ AI

- 规范成为团队契约，变更需要讨论
- 多人可以看彼此的 `workspace/{name}/journal.md` 了解进度
- 需要约定 commit message 格式（`feat(scope): xxx [ai-assisted]`）

### 模式 C：大团队 + AI 流水线

- 引入多 Agent（plan/implement/check/debug）
- CI 跑 AI Review（见 `ci/` 目录）
- task.json 的状态流程需要对接研发管理系统（TAPD/Jira）

## 七、效果度量

如何判断全 AI 开发环境有没有搭好？看这 4 个指标：

| 指标 | 不健康 | 健康 |
|------|--------|------|
| 新会话启动时间 | 5-10 分钟解释背景 | <30 秒（AI 读取 spec + journal） |
| 代码风格一致性 | 每次都要改格式 | AI 产出即达标 |
| 盲区泄漏率 | 周均 1-2 个 AI 引入 Bug | 月均 <1 个 |
| 规范沉淀速度 | 发现新约定只在对话说一次 | 24 小时内进入 Spec |

## 八、常见误解 Q&A

**Q：有了 AI 是不是不需要架构设计了？**
A：恰恰相反，AI 的产能让架构债务指数级放大。架构越清晰，AI 产出越高质量。

**Q：Spec 是不是应该尽可能详细？**
A：Spec 要「足够但最小」。过度详细会让 AI 读取成本高、维护难。发现 AI 产出不一致时再补，而不是预判。

**Q：多 Agent 是不是未来趋势，必须上？**
A：不是必须。单 Agent + 好的 Spec + 人把关，足够应对 95% 的业务开发。多 Agent 解决的是"长耗时任务"和"人力不足"，不解决"代码质量"。详见 `multi-agent-decision.md`。

**Q：会话日志要不要提交到 Git？**
A：**要**。日志是团队的集体记忆，新人/AI 都能从中学到决策脉络。用 `.gitignore` 过滤 `.developer` 身份文件即可。

**Q：Rule 文件写多少字合适？**
A：经验值：核心规则 300 行以内（AI 每次都加载），详细规范放 Spec 子文件（AI 按需读取）。超过 500 行的 Rule 往往加载即超预算。

## 九、一句话总结

> **让 AI 成为团队一员的办法，不是教 AI 懂你的项目，而是让项目对 AI 友好。**
> **规范、记忆、清单，三件套即可构建一个稳定、可演进的全 AI 开发环境。**
